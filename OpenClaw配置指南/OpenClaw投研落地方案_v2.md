# OpenClaw 投研场景落地方案（修订版）

> **核心场景**：研报阅读、投资分析、回测代码生成、数据处理、投研知识库  
> **用户群体**：10人买方投研团队  
> **关键决策**：API Token vs 包月订阅，哪个更划算？

---

## 一、投研场景使用模式分析

### 1.1 投研 vs 编程：使用模式差异

| 维度 | 编程场景（Coding Plan） | 投研场景（Research Plan） |
|------|------------------------|---------------------------|
| **使用频率** | 高频持续（写代码时一直开着） | 间歇爆发（看研报、写报告时用） |
| **会话长度** | 长（一个功能反复调试） | 超长（一份研报可能50万token） |
| **核心诉求** | 代码生成、调试、重构 | 信息提取、逻辑分析、报告撰写 |
| **工具形态** | IDE集成（Cursor/Claude Code） | 聊天+文档+代码混合 |
| **交互方式** | 文件级操作（读/写/执行） | 文档级操作（读/总结/对比） |

### 1.2 投研场景的典型使用画像

假设你们团队 10 人，其中：
- **基金经理 2人**：低频使用，主要是查数据、问结论
- **研究员 5人**：中高频，每天阅读 5-10 份研报，需要摘要和对比
- **量化/IT 3人**：高频，写回测代码、数据处理脚本

**日均使用量估算**：
```
研究员（5人）：
- 每份研报平均 5万字 = 约 7万 token
- 每天 5份 = 35万 token/人/天
- 加上追问、对比、重写 = 约 50万 token/人/天
- 5人 × 50万 = 250万 token/天

量化（3人）：
- 代码生成 + 调试 + 数据查询
- 约 20万 token/人/天
- 3人 × 20万 = 60万 token/天

合计：约 310万 token/天，月均 6600万 token
```

## 三、关键问题：包月服务能与 OpenClaw 集成吗？

### 3.1 残酷的现实

| 包月服务 | API 开放 | OpenClaw 集成难度 | 说明 |
|----------|----------|-------------------|------|
| Cursor Pro | ❌ 不开放 | 无法集成 | Cursor 是 IDE，不对外提供 API |
| Claude Code | ❌ 不开放 | 无法集成 | 仅限本地 CLI，无服务化接口 |
| Kimi Code | ⚠️ 有限 | 中等 | Kimi 有开放平台 API，但与 Kimi Code Pro 是两套体系 |
| ChatGPT Plus/Pro | ⚠️ 有限 | 中等 | API 与 Plus 是分开计费 |

**核心矛盾**：
- **包月服务**（Cursor、Claude Code、Kimi Code）是**终端产品**，面向个人用户，不开放 API
- **API Token** 是**基础设施**，面向开发者，可以集成到 OpenClaw

### 3.2 变通方案

如果你坚持用包月服务，有几种变通方式：

**方案1：人机混合模式**
```
简单/重复任务 → OpenClaw + API Token → 自动化
复杂/深度任务 → 人工使用 Cursor/Claude Code → 手动操作
```
- 缺点：工作流割裂，体验不一致

**方案2：浏览器自动化（Hack）**
```
OpenClaw 控制浏览器 → 登录 Cursor/Claude Code Web版 → 模拟人工操作
```
- 缺点：违反 ToS，封号风险，不稳定

**方案3：本地模型替代**
```
部署开源模型（Qwen2.5-72B/DeepSeek-R1-32B）在本地 → OpenClaw 直接调用
```
- 缺点：一次性硬件投入高（¥3-5万），维护成本

## 四、投研场景推荐方案

### 综合评估

| 维度 | API Token | 包月订阅 | 本地部署 |
|------|-----------|----------|----------|
| OpenClaw集成 | ✅ 完美 | ❌ 困难 | ✅ 可以 |
| 费用可控性 | ⚠️ 一般 | ✅ 好 | ✅ 好 |
| 长文本支持 | ⚠️ 贵 | ✅ 无限 | ✅ 无限 |
| 中文投研 | ✅ Kimi好 | ✅ Kimi好 | ⚠️ 一般 |
| 代码能力 | ⚠️ 一般 | ✅ Cursor好 | ⚠️ 一般 |
| 运维成本 | ✅ 低 | ✅ 低 | ⚠️ 高 |

### 推荐方案：API Token + 分层模型策略

```yaml
# 核心思路：用 OpenClaw 做智能路由，不同任务用不同模型

models:
  # 日常问答（便宜、快）
  default:
    provider: deepseek
    model: deepseek-chat
    
  # 研报摘要（长文本）
  research:
    provider: moonshot
    model: kimi-k2-5
    max_tokens: 200000  # 200万字符上下文
    
  # 深度分析（推理强）
  analysis:
    provider: deepseek
    model: deepseek-reasoner
    
  # 代码生成（代码强）
  coding:
    provider: anthropic
    model: claude-3-5-sonnet
    
  # 数据查询（结构化输出）
  data:
    provider: deepseek
    model: deepseek-chat
    response_format: json

# 智能路由规则
routing:
  - pattern: ["研报", "报告", "摘要", "总结"]
    model: research
    
  - pattern: ["分析", "对比", "逻辑", "推理"]
    model: analysis
    
  - pattern: ["代码", "策略", "回测", "python"]
    model: coding
    
  - pattern: ["数据", "查询", "表格", "json"]
    model: data
```

### 费用控制策略

**1. 设置预算上限**
```yaml
budget:
  daily_limit: ¥200      # 单日上限
  monthly_limit: ¥4000   # 单月上限
  alert_threshold: 80%   # 80%时告警
  
  overflow_action: notify  # 超限时通知，不阻断
```

**2. 长文本压缩**
```yaml
# 研报先过一遍预压缩，再送大模型
preprocessing:
  enabled: true
  
  # 用便宜模型先提取关键段落
  extract_key_sections:
    model: deepseek-chat  # 便宜
    prompt: "提取这份研报的投资评级、核心观点、关键数据、风险提示"
    
  # 只送关键段落给贵模型
  forward_to_main_model: true
```

**3. 缓存机制**
```yaml
# 相同研报不重复分析
cache:
  enabled: true
  ttl: 7d  # 缓存7天
  key: file_hash  # 按文件哈希去重
```

## 六、回答你的第二个问题

### 为什么我第一反应是 Token 而非包月？

1. **思维惯性**：我平时接触 Cursor/Claude Code 多，但它们是封闭产品，潜意识里知道没法集成
2. **角色定位**：OpenClaw 是"基础设施"，自然联想到 API 这种"基础设施级"的接入方式
3. **包月产品的尴尬**：Cursor/Kimi Code 的包月服务是**终端产品**，不是**基础设施**，和 OpenClaw 的定位冲突

### 包月服务真的不适合吗？

**适合个人，不适合 OpenClaw 集成场景**。

如果你们：
- **不用 OpenClaw**，直接用 Cursor + Kimi Code → 包月更划算
- **用 OpenClaw** 做统一入口 → 只能走 API Token

**混合模式可能是答案**：
- OpenClaw 处理 70% 的常规任务（API Token）
- 复杂代码任务交给 Cursor Pro（个人订阅）
- 长文本研报交给 Kimi Pro（个人订阅）

这样既能享受包月的不限量，又能保留 OpenClaw 的自动化能力。
